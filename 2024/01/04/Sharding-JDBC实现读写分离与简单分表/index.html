MySQL用Sharding-JDBC实现读写分离与简单分表

发表于2024-01-04|更新于2024-01-05|技术实践
|字数总计:1.6k|阅读时长:6分钟|阅读量:

ShardingSphere是一套开源的分布式数据库中间件解决方案组成的生态圈，它主要由Sharding-JDBC、Sharding-Proxy等相互独立的产品组成。他们均提供标准化的数据分片、分布式事务和数据库治理功能，可适用于如Java同构、异构语言、容器、云原生等各种多样化的应用场景。

Sharding-JDBC
定位为轻量级Java框架，在Java的JDBC层提供的额外服务。它使用客户端直连数据库，以jar包形式提供服务，无需额外部署和依赖，可理解为增强版的JDBC驱动，完全兼容JDBC和各种ORM框架。

适用于任何基于Java的ORM框架，如：JPA, Hibernate, Mybatis, Spring JDBC Template或直接使用JDBC。基于任何第三方的数据库连接池，如：DBCP, C3P0, BoneCP, Druid, HikariCP等。

支持任意实现JDBC规范的数据库。目前支持MySQL，Oracle，SQLServer和PostgreSQL。

读写分离

  • mysql的主从复制环境

    1. 配置主服务器（Master）：

      • 打开主服务器的配置文件（通常是 my.cnf 或 my.ini）。

      • 在文件中找到[mysqld]部分，并添加以下配置：

        • server-id：给主服务器分配一个唯一的标识符。每个服务器都必须有不同的 ID。
        • log-bin：启用二进制日志，用于记录主服务器上的所有更改。
        • binlog-format：设置二进制日志的格式。ROW 格式通常是推荐的选择，因为它可以提供更准确的复制。MySQL8以后默认为ROW，因此无需配置

    2. 重启主服务器：

      • 保存并关闭主服务器的配置文件。
      • 重启主服务器，使配置更改生效。

    3. 配置从服务器（Slave）：

      • 打开从服务器的配置文件。

      • 在文件中找到[mysqld]部分，并添加以下配置：

        • server-id：给从服务器分配一个唯一的标识符。每个服务器都必须有不同的 ID。
        • relay-log：设置从服务器上的中继日志文件的名称。
        • log-slave-updates：启用从服务器记录自己的二进制日志，以便其他从服务器可以复制它。
        • read-only：将从服务器设置为只读模式，以防止在从服务器上进行写操作。

    4. 重启从服务器：

      • 保存并关闭从服务器的配置文件。
      • 重启从服务器，使配置更改生效。

    5. 在主服务器上创建用于复制的用户：

      • 在主服务器上登录 MySQL。

      • 创建一个用于复制的用户，并为其授予复制相关的权限。例如：

        • 'replication_user'@'slave_ip'：替换为实际的从服务器 IP 地址和用户名。
        • 'password'：替换为实际的密码。

    6. 在从服务器上配置复制：

      • 在从服务器上登录 MySQL。

      • 运行以下命令配置复制：

        • 'master_ip'：替换为实际的主服务器 IP 地址。
        • 'replication_user' 和 'password'：替换为在主服务器上创建的复制用户的凭据。
        • 'mysql-bin.XXXXXX' 和 XXXXXX：替换为主服务器上当前二进制日志文件的名称和位置，可以使用 SHOW MASTER STATUS; 命令在主服务器上获取。

官方FAQ：为什么配置了某个数据连接池的spring-boot-starter（比如druid）和sharding-jdbc-spring-boot-starter时，系统启动会报错？

  • 因为数据连接池的starter（比如druid）可能会先加载并且其创建一个默认数据源，这将会使得sharding-jdbc创建数据源时发生冲突。
    解决办法为，去掉数据连接池的starter即可（保留主类依赖），sharing-jdbc自己会创建数据连接池。
